# Issue Reference Enforcement

Why: Ensures every merge is traceable to an issue, enabling audit trails, changelog generation, and accountability.

## Why CI Is the Source of Truth

Local hooks (commit-msg, pre-push) are **advisory only** — developers can bypass them with `--no-verify`. The CI pipeline is the single enforcement point because:

1. **Cannot be bypassed**: Runs server-side on every push and PR
2. **Consistent**: Same rules for all contributors regardless of local setup
3. **Auditable**: Check results are visible on the PR and in CI logs
4. **Branch-protection backed**: GitHub/GitLab can require the check to pass before merge

Local hooks are still useful as fast feedback, but they are **not** the policy enforcement mechanism.

## Accepted Issue Reference Patterns

**Regex**: `(#\d+|[A-Z][A-Z0-9]+-\d+)`

| Pattern | Example | Tracker |
|---------|---------|---------|
| `#<number>` | `#123`, `#42` | GitHub / GitLab |
| `ABC-123` | `KIH-42`, `PROJ-1001` | Jira |

References can appear anywhere in the commit message (subject line or body).

## Exceptions

- **Merge commits** are excluded from validation (auto-generated by git, not developer-authored)
- **Squash-merge workflows**: If `--check-pr-title` is used and the PR title contains a valid reference, all commits in the PR pass regardless of individual messages. This supports teams that rely on PR titles as the final squash commit message.

## Branch Protection Setup

### GitHub

1. Go to **Settings > Branches > Branch protection rules**
2. Select the target branch (e.g., `main`)
3. Enable **Require status checks to pass before merging**
4. Search for and add the check name: `issue-ref-check`
5. Enable **Require branches to be up to date before merging** (recommended)

### GitLab

1. Go to **Settings > Repository > Protected branches**
2. Set `main` as protected with appropriate merge/push permissions
3. Go to **Settings > Merge requests**
4. Under **Merge checks**, enable **Pipelines must succeed**
5. The `issue-ref-check` job in `.gitlab-ci.yml` will be required to pass

### Example Check Name

Use `issue-ref-check` as the status check / CI job name across all providers for consistency.

## GitHub Actions Workflow Snippet

```yaml
# .github/workflows/issue-ref-check.yml
name: Issue Reference Check

on:
  pull_request:
    branches: [main]

jobs:
  issue-ref-check:
    name: issue-ref-check
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0

      - name: Verify issue references in commits
        run: |
          chmod +x ./scripts/verify-issue-ref.sh
          ./scripts/verify-issue-ref.sh \
            --base "${{ github.event.pull_request.base.sha }}" \
            --head "${{ github.event.pull_request.head.sha }}" \
            --check-pr-title "${{ github.event.pull_request.title }}"
```

> **Note**: Copy `scripts/verify-issue-ref.sh` from the skill directory into the repository, or adapt the path to where your team stores CI scripts.

## GitLab CI Snippet

```yaml
# In .gitlab-ci.yml
issue-ref-check:
  stage: test
  script:
    - chmod +x ./scripts/verify-issue-ref.sh
    - ./scripts/verify-issue-ref.sh
        --base "$CI_MERGE_REQUEST_DIFF_BASE_SHA"
        --head "$CI_COMMIT_SHA"
        --check-pr-title "$CI_MERGE_REQUEST_TITLE"
  rules:
    - if: $CI_MERGE_REQUEST_IID
```

## Failure Recovery Playbook

When the `issue-ref-check` fails, follow these steps:

### 1. Identify offending commits

Look at the structured output:
```
ISSUE_REF_CHECK_BEGIN
STATUS=fail
TOTAL_COMMITS=5
MISSING_COUNT=2
MISSING_COMMITS=abc123def456,789012fed345
DETECTED_KEYS=#42,KIH-7
ISSUE_REF_CHECK_END
```

### 2. Fix with interactive rebase

```bash
# Reword commits that are missing issue refs
git rebase -i <base-sha>
# Mark offending commits as "reword"
# Add the issue reference to each commit message, e.g.:
#   [#42] Fix validation logic
# Force-push the branch
git push --force-with-lease
```

### 3. Alternative: fix via PR title (squash workflows only)

If your team uses squash merges and the CI check uses `--check-pr-title`:
- Edit the PR/MR title to include the issue reference (e.g., `[#42] Feature description`)
- Re-run the CI check

### 4. Alternative: amend the latest commit

If only the most recent commit is missing a reference:
```bash
git commit --amend -m "[#42] Original message here"
git push --force-with-lease
```

### 5. Re-run CI

After fixing, push the updated branch. The CI check will re-run automatically.
